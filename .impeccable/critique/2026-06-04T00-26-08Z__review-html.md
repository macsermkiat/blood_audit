## Anti-Patterns Verdict

**LLM assessment**: This does not read as AI-generated. The design is a functional clinical data dump rendered with project-appropriate OKLCH tokens from DESIGN.md. No gradient text, no glassmorphism, no hero metrics, no identical card grids. The tells of quick generation are subtler: the over-broad `td:nth-child(3,4)` CSS selector that misapplies monospace styling everywhere, and the `.vbox.llm` warm-yellow background (`oklch(96% 0.03 70)`) that's off-system. ## Overall Impression

Technically correct design execution that stops well short of being a useful review tool. The OKLCH tokens are applied consistently, the typography scale matches DESIGN.md, and the data is all present. What's missing is any hierarchy between signal (verdict comparison) and noise (raw HOSxP export rows). A reviewer sitting in a QI committee meeting opens this page and is greeted with the summary table — good — then immediately descends into Case 1's 7 open `<details>` sections. The verdict they need to discuss is three screens of scrolling away from the raw lab data that explains it. The single biggest opportunity: make the verdict boxes the clear primary view and demote source data to secondary.

## What's Working

**1. Token fidelity.** The `:root` variables are verbatim from DESIGN.md — OKLCH neutrals tinted toward hue 28, the exact accent value, the pill palette. The system font stack with Noto Sans Thai in the middle correctly handles mixed-language paragraphs. This is the right foundation.

**2. The `.verdict` grid layout.** Placing deterministic and LLM verdicts side-by-side in a `grid-template-columns: 1fr 1fr` box is exactly the right affordance for this comparison task. The pill badges inside each box make the verdict legible at a glance. This is the clearest design decision on the page.

**3. Progressive disclosure with `<details>`.** Using native `<details>` for source data sections is semantically correct, keyboard-accessible, and avoids JavaScript. The distinction between `details` (collapsed default) for medications and `details open` for diagnoses shows intentional default-state management, even if the defaults aren't calibrated well.

## Priority Issues

**[P1] The sticky nav encodes only case number and REQNO — no verdict signal**
- **Why it matters**: 7 of the 20 cases have a mismatch between deterministic and LLM verdicts (cases 1, 7, 10, 12, 14, 17, 20). Two are major mismatches (Potentially inappropriate vs. Appropriate: cases 7, 10). A committee reviewer cannot identify these from the nav and must read the summary table, identify the mismatch, then navigate. The nav adds no value over Ctrl+G.
- **Fix**: Append a short verdict-mismatch indicator to nav links. Example: `#7 68066907 [!]` for mismatches, `#10 68051474 [!]` for major ones. Color or a pill suffix would work. Even a compact legend key in the nav itself would help. Alternatively, bold the nav links for mismatch cases.
- **Suggested command**: `/impeccable clarify @/tmp/bba_mini/review.html`

**[P1] Multiple `<details open>` sections per case make source data the dominant visual**
- **Why it matters**: Each case renders 3–5 open detail sections (Diagnoses, ANC, Procedures, Progress notes) by default. For Case 2, the open Procedures section shows 5 rows, Progress notes shows 3 multi-paragraph notes. Over 20 cases, this pushes each verdict comparison below dozens of screens of raw data. The audit verdict is what the committee needs to discuss; source data is supporting evidence.
- **Fix**: Change all `<details>` to collapsed by default except perhaps the closest Hb history (the single most relevant data point for the threshold decision). Add a JS-free "expand-all source data" anchor at the top of each case section. Keep reasoning sections collapsed by default with a prominent "View reasoning" label.
- **Suggested command**: `/impeccable layout @/tmp/bba_mini/review.html`

**[P2] Summary table renders verdict columns as plain text — no pill styling**
- **Why it matters**: The legend at the top defines five verdict states by colored pill. The summary table — the only place all 20 verdicts appear together — renders them as plain text: "Needs review", "Potentially inappropriate". Color-blind or fast-scanning reviewers cannot process the table in under 10 seconds. Scanning for mismatches (different colors in Det vs. LLM columns) is impossible.
- **Fix**: Apply `.cls.cls-{verdict}` inline or wrap the verdict text in span elements with the existing pill CSS classes. No new CSS needed — the classes already exist. Also highlight rows where Det ≠ LLM with a faint warn-background tint on the `<tr>`.
- **Suggested command**: `/impeccable colorize @/tmp/bba_mini/review.html`

**[P2] The `.vbox.llm` warm-yellow tint (`oklch(96% 0.03 70)`) is off-system and semantically misleading**
- **Why it matters**: The yellow-green background with amber border (`oklch(80% 0.08 70)`) is warn-hue — it matches the `warn-bg` and `warn-fg` values in DESIGN.md. A reviewer's eye reads yellow as "caution/warning." The LLM verdict box isn't a warning; it's a parallel classification. The color distinction is needed (it's hard to parse det vs. LLM without it) but the hue choice is wrong.
- **Fix**: Replace with a neutral raised surface plus a distinct left-side 3px accent strip in a different hue — or use the info palette: `oklch(94% 0.04 280)` / `oklch(80% 0.08 280)` (info-bg/border, purplish-blue, not in warm-red territory or warn-yellow). The `--info-bg` and `--info-fg` variables are already in the `:root` block.
- **Suggested command**: `/impeccable colorize @/tmp/bba_mini/review.html`

**[P2] `td:nth-child(3), td:nth-child(4)` monospace/muted rule applies to all tables globally**
- **Why it matters**: The rule is designed to style code-identity columns (Source IDs, confidence scores) in the LLM indication table. It misapplies to the diagnostic table's "Type" and "ICD10WHO" columns, the procedures table's "OPRTACT" and "INCOME" columns, and others — rendering human-readable text in tiny (0.75rem), muted, monospace type. "Principal Diagnosis" at 0.75rem monospace is illegible at normal reading distance.
- **Fix**: Scope the rule to `.ind td:nth-child(3), .ind td:nth-child(4)` (the indication table class). Remove the global selector. The indication table already has `class='ind'`.
- **Suggested command**: `/impeccable harden @/tmp/bba_mini/review.html`

## Minor Observations

- The `h5` element (used for "Indications", "Negative evidence" inside verdict boxes) is the same size as body text and differentiated only by weight and muted color — minimal hierarchy.
- The "Returned" column in the summary table is all "—" (em-dash placeholder). Either remove the column until data exists, or label it "Blood return (not tracked)" so reviewers know why it's empty.
- No `@media print` styles. The sticky nav will print on every page, and `<details open>` content may not print as expected in all browsers. This document will likely be printed for committee distribution.
- The body `max-width: 1200px` is appropriate for the wide tables but makes the prose descriptions (LLM rationale, negative evidence lists) uncomfortably wide at full viewport — consider capping `.reasoning` at 80ch.
- The `<html lang="en">` is correct but Thai content inside `<details lang='th'>` should cascade to `<pre>` children — verify browser behavior for `lang` inheritance in detail elements.
