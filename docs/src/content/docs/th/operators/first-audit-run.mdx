:::note[ขอบเขต Phase 1]
ใน Phase 1, `bba audit` ทำเฉพาะ **ส่วน ingest** ของ pipeline
แถว classification (`APPROPRIATE`, `INAPPROPRIATE`, `NEEDS_REVIEW`, ฯลฯ)
ยังไม่ถูกเขียนโดยคำสั่งนี้ ส่วน LLM analysis จะเชื่อมต่อใน Phase 1.5
ดูรายละเอียดที่ [คำสั่ง audit](/blood_audit/th/operators/audit-command/)
:::

## 1. ทางเลือก: ตรวจชุดข้อมูลก่อน

<CliCommandCard
  command="bba ingest INPUT_CSV --schema-version v1"
  summary="ตรวจสอบไฟล์ CSV จาก HOSxP, hash input, drain rows ผ่าน normalization และเขียน completion marker ใต้ $BBA_DATA_DIR/ingest/v1"
  flags={[
    { name: "INPUT_CSV", description: "เส้นทางของ CSV หลักจาก HOSxP", required: true },
    { name: "--schema-version v1", description: "schema fingerprint ที่ใช้ร่วมในการสร้าง run_id" },
  ]}
  envVars={["BBA_DATA_DIR"]}
  example={`uv run bba ingest /var/data/exports/PHI_xxx_001.csv --schema-version v1`}

ขั้นตอนนี้เหมาะเมื่ออยากตรวจ bundle แยกก่อนรัน audit สำหรับเส้นทางปกติของ
operator ใน Phase 1 สามารถรัน `bba audit --input ...` ได้เลย เพราะคำสั่ง audit
จะเรียก ingest validation/drain ภายในอยู่แล้ว

## 2. รัน audit (Phase 1: ส่วน ingest)

<CliCommandCard
  command="bba audit --input INPUT_CSV"

  flags={[

คำสั่งนี้รันซ้ำได้อย่างปลอดภัย: ถ้าใช้ CSV เดิม ระบบจะไม่เขียนผลซ้ำ
ยกเว้นใส่ `--force` ซึ่งจะบันทึก idempotency override ไว้ใน audit trail

## 3. ตรวจสอบ run markers

Subcommand 4 ตัวที่ใช้ดูผลหลัง audit (`evaluate`, `report`,
`serve-dashboard`, `sentinel`) จะ raise `CliError` ใน Phase 1 โดยตั้งใจ ดู
[Integration seams](/blood_audit/th/operators/integration-seams/)

ตอนนี้สิ่งที่ตรวจสอบได้ดีที่สุดคือ run markers และ structured logs:

```bash
# ตรวจว่ามี directory ของ run_id และ run-state marker
ls -la "$BBA_DATA_DIR/audit/<RUN_ID>/"
ls "$BBA_DATA_DIR/audit_runs/<RUN_ID>/"
```

## สิ่งที่จะเห็นใน Phase 1.5

เมื่อเชื่อม LLM analysis แล้ว `bba audit` จะเขียน `AuditRow`
1 แถวต่อคำสั่งให้เลือดที่ตรวจสอบลงใน audit store เพิ่มเติม
คอลัมน์ `final_classification` จะมีค่าเป็น 1 ใน verdict หลัก 5 ค่า —
`APPROPRIATE`, `INAPPROPRIATE`, `NEEDS_REVIEW`,
`INSUFFICIENT_EVIDENCE`, `POTENTIALLY_INAPPROPRIATE` — หรือเป็น terminal
ฝั่ง routing/reservation: `RETURNED_NOT_TRANSFUSED` และ
`PERIOP_TRANSFUSION_EXEMPT` (ไม่นับในการให้คะแนน),
`PREOP_OVER_RESERVATION` และ `PREOP_RESERVATION_UNCONFIRMED`
(มี flag ควบคุม, default-off) ส่วน `LLM_REVIEW` เป็นแนวคิดการ routing
(ไม่ใช่ค่าที่บันทึกจริง) ถ้า quote-grounding ล้มเหลวจะกลายเป็น
`final_classification = NEEDS_REVIEW` พร้อม `review_reason` ที่ระบุ
