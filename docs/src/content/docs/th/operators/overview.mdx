## เอกสารส่วนนี้สำหรับใคร

คุณกำลังดูแล blood_audit ในนามทีม KCMH IT งานของคุณมี 4 ขั้นตอน:

1. ติดตั้ง package
2. ตั้งค่า environment variables ที่จำเป็น
3. รัน `bba audit --input ...`; ใช้ `bba ingest` ก่อนเฉพาะเมื่อต้องการ
   preflight ตรวจ bundle แยกต่างหาก
4. ส่งชุดข้อมูลออดิทให้คณะกรรมการคุณภาพ (Quality Improvement, QI)

ส่วน subcommand อีก 4 ตัว คือ `evaluate`, `report`, `serve-dashboard`
และ `sentinel` จะถูกเชื่อมต่อโดยทีม deployment ในภายหลัง หากต้องการทราบว่า
ทำไมตอนนี้คำสั่งเหล่านี้จึง fail โดยตั้งใจ ดูได้ที่
[Integration seams](/blood_audit/th/operators/integration-seams/)

## สิ่งที่ต้องเตรียมก่อนเริ่ม

สำหรับ Phase 1 ตอนนี้:

- **Python 3.13+** พร้อม [uv](https://github.com/astral-sh/uv) เวอร์ชัน ≥ 0.9.24
- **ไดเรกทอรีข้อมูลถาวร** ที่ service user เขียนได้สำหรับ run-state markers
- **ชุดข้อมูลส่งออกจาก HOSxP** — ดูรายละเอียดที่ [Ingest CSV](/blood_audit/th/operators/ingest-csv/)

สำหรับ Phase 1.5 เมื่อเชื่อม analysis/reporting:

- **PostgreSQL 14+** สำหรับ `batch_runs` และ review-action state
  ต้องใช้ DSN ที่มี driver suffix `+psycopg` (psycopg3 ไม่ใช่ psycopg2)
- **Anthropic API key** ที่เปิดใช้งาน Batch API ได้

## ผลลัพธ์เมื่อรันสำเร็จ

1. `bba ingest <csv>` ตรวจสอบชุดข้อมูล คำนวณ hash และเขียน ingest completion marker ใต้ `$BBA_DATA_DIR/ingest/v1/`
2. `bba audit --input <csv>` ใน Phase 1 จะ validate/drain bundle ภายใต้
   `$BBA_DATA_DIR/audit/<run_id>/` และบันทึก run-state/compliance marker
   ส่วนการเขียน `AuditRow` 1 แถวต่อคำสั่งให้เลือดที่ audit และการติดตาม
   `batch_runs`/review state จะเชื่อมใน Phase 1.5
3. การรันซ้ำด้วย input เดิมจะไม่ทำอะไรใหม่ ยกเว้นใส่ `--force`
   (รายละเอียดดู [Run-level idempotency](/blood_audit/th/operators/troubleshooting/#run-level-idempotency))

## สิ่งที่คาดหวังได้

- ตัวจำแนก deterministic จะทำงานก่อนเสมอ เมื่อเชื่อม Phase 1.5 แล้ว
  แถว `NEEDS_REVIEW` และ `POTENTIALLY_INAPPROPRIATE` จะเข้าสู่เส้นทาง LLM
  ซึ่งอาจใช้เวลาหลายนาทีกว่าจะเสร็จ
- มี exclusion แบบเด็ดขาดสำหรับผลิตภัณฑ์ที่ไม่ใช่ RBC, ผู้ป่วยอายุ < 15 ปี,
  และ ICD-10 conditions ที่อยู่นอก scope ส่วนคำสั่งที่ไม่มี Hb ก่อนให้เลือด
  จะถูกเก็บเป็น `INSUFFICIENT_EVIDENCE`
- PHI ทั้งหมดจะถูก de-identify ก่อนที่ข้อมูลใด ๆ จะออกจาก audit container
