## หลักการแก้ code

- ทำให้เล็กและตรงประเด็น
- ใช้ pattern เดิมของ repo ก่อนสร้าง abstraction ใหม่
- ถ้าแตะ behavior ให้เพิ่มหรือปรับ test ตามความเสี่ยง
- ห้าม log PHI จริง และห้ามใส่ PHI จริงใน fixture, issue หรือเอกสาร

## เอกสาร

เมื่อ concept ที่ผู้ใช้หรือ operator ต้องรู้เปลี่ยน ให้ update เอกสารคู่กัน
ทั้ง docs site และ `docs/CONTEXT.md` เพื่อให้ repo มี single source สำหรับ
handoff ระยะยาว
