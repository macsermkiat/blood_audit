## Section files

```
<output_dir>/
  hospital_trend.csv                          # committee
  ward_scorecard.csv                          # committee
  indication_distribution.csv                 # committee
  cohort_exception.csv                        # committee
  pipeline_health.csv                         # committee
  physician_own_view_<physician_id>.csv       # one file per requested physician
  report_<YYYY-MM-DD>.pdf                     # committee PDF
```

The PDF filename embeds the first-of-month date
(e.g. `report_2026-05-01.pdf`).

## Stored classifications (used by every section)

The `final_classification` column comes from the audit store and is one
of nine literals:

- `APPROPRIATE`
- `INAPPROPRIATE`
- `NEEDS_REVIEW`
- `INSUFFICIENT_EVIDENCE`
- `POTENTIALLY_INAPPROPRIATE` *(rule-only; not used by every report
  section)*
- `PREOP_OVER_RESERVATION` *(MSBOS over-reservation; scored as
  `INAPPROPRIATE` in the report aggregates)*
- `PREOP_RESERVATION_UNCONFIRMED` *(flag-gated, default-off)*
- `RETURNED_NOT_TRANSFUSED` *(returns terminal; **excluded from
  scoring**)*
- `PERIOP_TRANSFUSION_EXEMPT` *(returns / declared-use terminal;
  **excluded from scoring**)*

The `hospital_trend`, `ward_scorecard`, and `cohort_exception` aggregate
models additionally track an `over_reservation_count` field; the CSV
writer does not serialize it as a column yet, so the emitted CSV headers
are unchanged.

`LLM_REVIEW` is a routing concept inside the deterministic classifier;
it is not a stored value. Failed quote-grounding becomes
`final_classification = NEEDS_REVIEW` with `review_reason` set.

`INSUFFICIENT_EVIDENCE` is **never** folded into `INAPPROPRIATE` — PRD
contract: "documentation absence ≠ inappropriate". The
`pipeline_health` section keeps these counts in separate buckets.

## Reproducibility footer (every row, every PDF page)

| Column | Meaning |
| --- | --- |
| `policy_version` | KCMH PR-17.x policy revision driving the deterministic classifier. |
| `model_id` | Snapshot-pinned Anthropic model ID. |
| `redactor_version` | Version of `macsermkiat/thai-medical-deid`. |
| `redactor_model_sha` | SHA of the redactor's HuggingFace model checkpoint. |
| `prompt_hash` | SHA of the system prompt + few-shot bundle. |
| `evidence_bundle_hash` | SHA of the canonical evidence-bundle schema. |

## Month bucketing

`ReportInputs.month` is a `date` pinned to the first of the month. The
filter window is `[month_local_start, next_month_local_start)`
interpreted in Asia/Bangkok. An order at 23:00 Bangkok on the 31st
belongs to its local month, not to the UTC-month it happens to fall in.

## How this is produced

`bba.report_generator.generate_monthly_report(inputs: ReportInputs)`. The
CLI wrapper (`bba report --run-id <id>`) is **wired**: it sources
`ReportInputs` from the audit store, resolves ward/physician attribution
from `BBA_BDVST_CSV` / `BBA_DCT_CSV`, and writes the committee CSVs,
per-physician CSVs, and PDF under `$BBA_DATA_DIR/reports/<run_id>/`.
