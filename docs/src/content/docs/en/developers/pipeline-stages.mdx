:::note[What runs today via `bba audit`]
In Phase 1, `bba audit` wraps **stage 1 only** (ingest). Stages 2–12 are
implemented as standalone modules with their own tests, but the
orchestration glue in `bba.audit_pipeline.run_pipeline` is wired through
the CLI in Phase 1.5. The descriptions below cover what each module
does; the CLI-level scope is documented on
[audit command](/blood_audit/en/operators/audit-command/).
:::

## Stage-by-stage

### 1. `bba.ingest`

Reads the HOSxP CSV bundle. Validates against the published schema,
computes `input_csv_hash`, streams rows through normalization, and writes
the run completion marker. Idempotent on the file content.

### 2. `bba.audit_orders`

Filters the ingested orders down to auditable RBC and platelet
transfusion orders (the `component_map` intake gate routes each order to
its component family). Hard exclusions: age < 15, out-of-scope products
(LPRC, LDPRC, SDR), ICD-10 gates. Resolves the per-order anchor
datetime.

### 3. `bba.hb_lookup`

For each surviving order, finds the most recent pre-transfusion Hb result
in the Lab table within a 7-day lookback, prefers HEMATOLOGY over POCT,
and computes delta-Hb windows for acute-blood-loss evidence.

### 4. `bba.vitals_extractor`

Extracts BP, HR, urine output, and temperature from MED + progress notes
around the anchor datetime. Outputs structured vitals records.

### 5. `bba.cohort_detector`

Assigns the order to a deterministic cohort such as default,
cardiac-surgery, ortho-surgery, ESRD-EPO, cardiopulmonary-comorbidity
(heart disease), MTP, active heme malignancy, or unknown. The cohort drives
the Hb threshold or review route in the deterministic classifier.

### 6. `bba.deterministic_classifier`

The 3-tier engine. See [3-tier Hb classifier](/blood_audit/en/developers/three-tier-hb/)
for the full rule. Three default-ON overlay arms act at this stage:
the returns ledger (`bba.returns_ledger`) can terminal the order as
`RETURNED_NOT_TRANSFUSED`, declared-use USETYPE routing
(`bba.declared_use`) as `PERIOP_TRANSFUSION_EXEMPT`, and the MSBOS
reservation audit (`bba.preop_reservation`) can surface
`PREOP_OVER_RESERVATION` for planned-operation over-reservation.

### 7. `bba.evidence_bundle_builder`

Builds the unredacted evidence bundle for the LLM context window: lab
values, vitals, anchor notes, medications, cohort evidence, ICD codes,
and procedure timing.

### 8. `bba.deid_redactor`

De-identifies the bundle. Installs a PHI-scrubbing `sys.excepthook` so any
uncaught exception cannot leak PHI to logs. Reuses
`bba.deid_redactor.PHI_REGEXES` — never duplicate that pattern set.

### 9. `bba.prompt_builder`

Renders the few-shot + clinical-history system message. Stable across runs
for prompt-caching effectiveness.

### 10. `bba.llm_client`

Submits the Batch API request. Sonnet → Opus escalation; prompt caching
on; structured-output schema validation. Resumable: on restart, re-queries
`batch_runs` rows in `pending` / `submitted` / `partial` states.

### 11. `bba.quote_grounder`

Runs the [6 fail-closed grounding layers](/blood_audit/en/developers/quote-grounding/).
Failures set `final_classification = NEEDS_REVIEW` with `review_reason =
"hallucination_suspect"`. Grounding never silently passes.

### 12. `bba.audit_store`

Persists one `AuditRow` per audited order (and one `LlmCall` per
Anthropic call) immutably to Parquet. Datetimes are tz-aware UTC at the
model boundary; nested JSON fields are deeply frozen. See
[Audit store](/blood_audit/en/developers/audit-store/) for the
classification literals and the Phase 1 vs 1.5 split.

## State machine for LLM-route rows

Each row routed to the LLM carries a `batch_runs` state:

```text
pending → submitted → partial → complete
                              ↘ failed
```

A crashed process is resumable from any non-terminal state without manual
intervention. `failed` is terminal and requires investigation.
