## What blood_audit does

blood_audit reads completed HOSxP exports from the hospital information
system. The `component_map` intake gate routes each blood-bank order to its
component family, then the analysis leg compares each audited red blood cell
(RBC) order against the 3-tier hemoglobin (Hb) policy and each platelet order
against the §5.1 count gate, and writes one audit row per audited order to the
audit store. The RBC classifier checks Lab Hb, ICD-10 diagnoses, ICD-9-CM
procedures, medications, blood-order patterns, and procedure timing in a fixed
order. This analysis leg runs today through the pilot scripts and is wired into
the CLI in Phase 1.5.

Before any Hb judgment, three default-ON routing arms may terminal an order
out of appropriateness scoring: the **returns ledger** (per-unit BDVSTTRANS
disposition), **declared-use USETYPE routing** (order-level surgical /
type-and-screen intent), and the **MSBOS pre-op reservation audit**
(over-reservation screening for planned operations).

The final `final_classification` stored for each transfusion is one of:

- `APPROPRIATE` — the order matched policy.
- `INAPPROPRIATE` — the order did not match policy.
- `POTENTIALLY_INAPPROPRIATE` — the deterministic rule flagged the order
  but did not promote it to LLM review.
- `NEEDS_REVIEW` — the LLM ran but its answer requires a human reviewer
  (uncertain reasoning, failed quote-grounding, or low confidence).
- `INSUFFICIENT_EVIDENCE` — documented evidence was missing; the row is
  *not* `INAPPROPRIATE`, by design.
- `RETURNED_NOT_TRANSFUSED` — every unit was returned un-transfused
  (returns terminal; **excluded from scoring**).
- `PERIOP_TRANSFUSION_EXEMPT` — declared surgical / type-and-screen use or
  peri-op transfusion (returns terminal; **excluded from scoring**).
- `PREOP_OVER_RESERVATION` — reserved units exceeded the MSBOS
  recommendation for the planned operation (counts as an over-reservation
  finding, not an Hb verdict).
- `PREOP_RESERVATION_UNCONFIRMED` — reserve-ahead order whose transfusion
  could not be confirmed (behind a default-off flag).

Rows that the deterministic classifier routes to the LLM go through six
fail-closed quote-grounding layers; a failed grounding becomes
`NEEDS_REVIEW`, never a published result.

## Phase 1 status

`bba audit` today wraps the **ingest leg** of the pipeline. It validates the
HOSxP CSV bundle, streams rows through normalization, and records run-state
markers under `$BBA_DATA_DIR/audit/<run_id>/` and
`$BBA_DATA_DIR/audit_runs/<run_id>/` with idempotent run-level guarantees.
In these docs, a "cohort" means the clinical-context bucket for one audit
row, such as default, cardiac surgery, orthopedic surgery, ESRD with
dialysis-medication evidence, MTP, active heme malignancy, cardiopulmonary
comorbidity (heart disease), or unknown context.

The LLM-driven analysis leg (deterministic classifier → evidence bundle
→ de-id → prompt → batch → quote grounding → audit-store write) is
implemented as separate modules but is **wired in Phase 1.5**. See
[audit command](/blood_audit/en/operators/audit-command/) for what runs today.

Six fail-closed quote-grounding layers protect every LLM review row from
hallucination. The deterministic classifier remains the authoritative source.

## Phase 2 status — platelet auditor (merged)

The platelet auditor is **merged** (PR #85). The `component_map` intake gate
tells a platelet order from a red-cell one, and the deterministic platelet leg
(`platelet_lookup` + `platelet_classifier` §5.1 gate) is live. It auto-clears
nothing: a count ≥ 100 ×10³/µL routes to review, a count below the ceiling
defers to the LLM, and a missing count returns `INSUFFICIENT_EVIDENCE`. The
platelet LLM leg sits behind the default-off `PLATELET_LLM_ENABLED` flag, and
the RBC path is byte-identical throughout.
