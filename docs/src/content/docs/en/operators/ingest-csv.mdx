`bba ingest` is the entry point for every audit. It validates, hashes, and
materializes the input bundle into a DuckDB + Parquet snapshot under
`$BBA_DATA_DIR`.

## Expected tables

The HOSxP export bundle contains roughly 10 tables. The audit pipeline
reads, at minimum:

- **Transfusion order header** — (HN, REQNO) pairs, product code, request
  datetime. Hard exclusions remove non-RBC products (LPRC, LDPRC, SDR).
- **Lab** — pre-transfusion hemoglobin results, keyed by HN and timestamp.
- **MED / progress notes** — text source for vital signs and acute-blood-loss
  indicators.
- **Operative table** — procedure code + timestamp (used for cohort
  detection and procedure-proximity logic).
- **ICD-10 / ICD-9-CM** — diagnosis and procedure code reference.

The exact column contract is defined in `src/bba/ingest/`.

## What ingest writes

For each successful run:

- A DuckDB database under `$BBA_DATA_DIR/duckdb/<run_id>.duckdb`.
- Parquet snapshots under `$BBA_DATA_DIR/parquet/<run_id>/`.
- A completion marker; absence of this marker means the run is incomplete
  and a re-ingest will start fresh.

## run_id derivation

```text
run_id = sha256(input_csv_hash + schema_fingerprint + code_version)
```

This is the key invariant: identical input + identical code = identical
`run_id`. The audit pipeline refuses to write a duplicate `(run_id, encounter_id)`
row to `audit_store` — see [Run-level idempotency](/blood_audit/en/operators/troubleshooting/#run-level-idempotency).

## Sample placeholder values

For examples in this site we use synthetic placeholders only:

```text
HN: PHI_xxx_001
AN: PHI_xxx_002
REQNO: PHI_xxx_003
```

Never substitute real HN / AN values in documentation, screenshots, or
issue reports.
