`bba ingest` is the entry point for validating a HOSxP export bundle. In
the current Phase 1 CLI it validates headers, hashes inputs, streams rows
through normalization, and writes the run completion marker under
`$BBA_DATA_DIR`. DuckDB + Parquet writers are the intended next storage
shape, but they are not emitted by `bba ingest` yet.

## Expected tables

The HOSxP export bundle contains 12 canonical tables:

| Table | Used for |
| --- | --- |
| `BDVST` | Transfusion order header: HN, AN, REQNO, product, request datetime, ordering doctor (DCTREQ). |
| `BDVSTDT` | Order detail rows and product/unit-level context. |
| `BDTYPE` | Blood-product reference. |
| `BDVSTST` | Transfusion-status reference. |
| `Diagnosis` | ICD-10 diagnosis evidence for exclusions and cohorts. |
| `Lab` | Hb, ANC, and other lab evidence. |
| `Med` | Medication evidence: dialysis meds, chemo meds, crystalloid volume. |
| `IPDADMPROGRESS` | Progress-note evidence for the LLM bundle. |
| `IPDNRFOCUSDT` | Nursing-note evidence for the LLM bundle. |
| `IPTSUMOPRT` | Operative events and procedure timestamps. |
| `INCPT` | Procedure/charge-adjacent event table validated as part of schema v1. |
| `ICD9CM` | Procedure-code dictionary and OR/non-OR flag. |

The exact column contract is defined in `src/bba/ingest/`.

## What ingest writes

For each successful `bba ingest` run today:

- `$BBA_DATA_DIR/ingest/v1/` is used as the ingest output root.
- A `_run_<run_id>.complete` marker records that the schema validation and
  row-normalization drain completed.
- Absence of this marker means the run is incomplete and a re-ingest will
  start fresh.

For `bba audit --input ...` in Phase 1, the same ingest validation/drain is
run under `$BBA_DATA_DIR/audit/<run_id>/`, and one run-state row is recorded
per validated HOSxP table.

## run_id derivation

```text
run_id = sha256(input_csv_hash + schema_fingerprint + code_version)
```

This is the key invariant: identical input + identical code = identical
`run_id`. The audit pipeline refuses to write a duplicate `(run_id, encounter_id)`
row to `audit_store` — see [Run-level idempotency](/blood_audit/en/operators/troubleshooting/#run-level-idempotency).

## Sample placeholder values

For examples in this site we use synthetic placeholders only:

```text
HN: PHI_xxx_001
AN: PHI_xxx_002
REQNO: PHI_xxx_003
```

Never substitute real HN / AN values in documentation, screenshots, or
issue reports.
