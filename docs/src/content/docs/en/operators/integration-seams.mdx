Three subcommands of `bba` raise `CliError` when invoked in Phase 1. This is
**by design**: the underlying modules ship and are tested, but the CLI
hand-off requires deployment-specific wiring (auth, stores, resolvers)
that does not belong in Phase 1. Do not "fix" them.

(`bba report` used to be the fourth seam; it is now **wired** — it
renders committee CSVs, per-physician CSVs, and the PDF from the audit
store, resolving ward/physician attribution from `BBA_BDVST_CSV` /
`BBA_DCT_CSV`.)

## Why fail loud instead of running with defaults?

Each of these seams touches PHI, billing, or clinician notification. A
silent default could leak data or alert the wrong people. The CLI surfaces
the missing contract explicitly so the deployer wires it deliberately.

## When will these be wired?

Phase 1.5, after the M8 pilot signs off. Tracked under the
deployment-readiness epic in the project repo, not as a code bug.
