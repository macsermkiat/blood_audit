This page walks the Phase 1 happy path: a HOSxP export goes in, the bundle
is validated and row-normalized, and idempotent run markers are written.

:::note[Phase 1 scope]
In Phase 1, `bba audit` runs the **ingest leg** only. Classification rows
(`APPROPRIATE`, `INAPPROPRIATE`, `NEEDS_REVIEW`, etc.) are not yet
written by this command; the LLM analysis leg is wired in Phase 1.5.
See [audit command](/blood_audit/en/operators/audit-command/) for the
exact contract.
:::

## 1. Optional: validate the export bundle

<CliCommandCard
  command="bba ingest INPUT_CSV --schema-version v1"
  summary="Validates the HOSxP CSV bundle, hashes inputs, drains rows through normalization, and writes a completion marker under $BBA_DATA_DIR/ingest/v1."
  flags={[
    { name: "INPUT_CSV", description: "Path to the primary HOSxP export CSV.", required: true },
    { name: "--schema-version v1", description: "Schema fingerprint to include in run_id derivation." },
  ]}
  envVars={["BBA_DATA_DIR"]}
  example={`uv run bba ingest /var/data/exports/PHI_xxx_001.csv --schema-version v1`}
/>

This step is useful when you want a separate preflight validation pass.
For the normal Phase 1 operator path, `bba audit --input ...` below also
runs the ingest validation/drain internally.

Under the hood, the `run_id` is `sha256(input_csv_hash +
schema_fingerprint + code_version)`. You do not need to compute it
yourself; the CLI does.

## 2. Run the audit (Phase 1: ingest leg)

<CliCommandCard
  command="bba audit --input INPUT_CSV"
  summary="Phase 1: validates and drains the HOSxP CSV bundle under $BBA_DATA_DIR/audit/<run_id>/ and records completion/run-state markers."
  flags={[
    { name: "--input INPUT_CSV", description: "Same CSV path used for ingest." },
    { name: "--run-id RUN_ID", description: "Reuse an existing run_id (no-op unless --force)." },
    { name: "--force", description: "Override idempotency. Writes a compliance row to audit_log with idempotency_override=true." },
  ]}
  envVars={["BBA_DATA_DIR"]}
  example={`uv run bba audit --input /var/data/exports/PHI_xxx_001.csv`}
/>

The command is **idempotent**. Re-running the same CSV is a no-op unless
`--force` is passed. Forced re-runs are recorded as idempotency overrides.

## 3. Inspect the run markers

The four post-audit subcommands (`evaluate`, `report`,
`serve-dashboard`, `sentinel`) raise `CliError` in Phase 1 — see
[Integration seams](/blood_audit/en/operators/integration-seams/).

For now, the most useful checks are on the run markers and logs:

```bash
# Confirm the audit run-state directory exists.
ls -la "$BBA_DATA_DIR/audit/<RUN_ID>/"
ls "$BBA_DATA_DIR/audit_runs/<RUN_ID>/"
```

## What you will see in Phase 1.5

Once the LLM analysis leg is wired, `bba audit` will additionally write
one `AuditRow` per audited order to the audit store. The
`final_classification` column holds one of the five core verdicts —
`APPROPRIATE`, `INAPPROPRIATE`, `NEEDS_REVIEW`,
`INSUFFICIENT_EVIDENCE`, `POTENTIALLY_INAPPROPRIATE` — or one of the
routing/reservation terminals: `RETURNED_NOT_TRANSFUSED` and
`PERIOP_TRANSFUSION_EXEMPT` (excluded from scoring),
`PREOP_OVER_RESERVATION`, and `PREOP_RESERVATION_UNCONFIRMED`
(flag-gated, default-off). `LLM_REVIEW` is a routing concept (not a
stored value); failed quote-grounding becomes
`final_classification = NEEDS_REVIEW` with `review_reason` set.
