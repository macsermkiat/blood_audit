`bba audit` is the only wired write path in Phase 1. It runs the full
12-stage pipeline, classifies each RBC transfusion, and writes audit rows
to Postgres. The four post-audit subcommands (`evaluate`, `report`,
`serve-dashboard`, `sentinel`) are [integration seams](/blood_audit/en/operators/integration-seams/).

## Either `--input` or `--run-id`, not both

Choose `--input` for first-time audits (ingest + audit in one step) or
`--run-id` to re-run audit against an existing snapshot.

## `--force` writes a compliance row

When `--force` is set, the pipeline writes an entry to `audit_log`
documenting **why** the re-run was forced. This preserves the audit trail
for the QI committee. Do not bypass this by editing the database directly.

## Batch API resume semantics

For each LLM_REVIEW transfusion, the system creates one tracking row in
the `batch_runs` table. Each row moves through these states:

- `pending` — waiting to be sent to the Anthropic Batch API.
- `submitted` — sent; waiting for a response.
- `partial` — some results returned; still polling.
- `complete` — all results returned and stored.
- `failed` — terminal; needs investigation.

If the audit process crashes part-way through, the next `bba audit` run
picks up from `submitted` and `partial` rows automatically. You will not
lose work.

## Safety boundaries

- The audit reads HOSxP exports only. All audit output is written to the
  audit database and audit files. HOSxP tables are never touched.
- The audit runs after the fact and stays silent. It does not send
  real-time alerts to clinicians. This is quality improvement (QI), not
  Software as a Medical Device (SaMD).
- The audit store rejects duplicate `(run_id, encounter_id)` writes.
  Re-runs of the same input always produce the same result, in the same
  place. This invariant lives in the store layer and stays that way.
