:::caution[Phase 1 scope]
In Phase 1, `bba audit` wraps the **ingest leg** of the pipeline. It
validates the HOSxP CSV bundle, streams rows through normalization under
`$BBA_DATA_DIR/audit/<run_id>/`, and records completion/run-state markers
for idempotency. The LLM-driven analysis leg (deterministic classifier →
evidence bundle → de-id → prompt → batch → quote ground → audit-store
write) is implemented as separate modules but is **wired in Phase 1.5**.
Until then, classification rows are not yet produced by this command.
:::

The other four subcommands (`evaluate`, `report`, `serve-dashboard`,
`sentinel`) are [integration seams](/blood_audit/en/operators/integration-seams/)
that fail loud by design.

## When to use `--input` vs `--run-id`

- `--input INPUT_CSV` — first run for this bundle. Computes a `run_id`
  from the input hash + schema fingerprint + code version, ingests, and
  marks the run complete.
- `--run-id RUN_ID` — re-attach to an already-completed run. With no
  `--force`, this is a no-op (the run is already complete). With
  `--force`, see below.

## `--force` writes a compliance row

When `--force` is set, the command writes an entry to `audit_log` with
`idempotency_override=true` documenting that the re-run was forced. This
preserves the audit trail for the QI committee. Do not bypass this by
editing the run-state files directly.

## Idempotency at the run level

Concurrent invocations on the same `run_id` are serialized via
`store.acquire_run_lock`. Two parallel CLI processes cannot both pass
the "already complete" guard and execute ingest twice. Concurrent
audits of *different* inputs run in parallel without contention.

## Phase 1.5 (preview)

When the LLM analysis leg is wired, this command will additionally:

- Build the evidence bundle, de-identify it, and submit the LLM_REVIEW
  rows to the Anthropic Batch API.
- Run the six fail-closed quote-grounding layers on every LLM response.
- Write one `AuditRow` per audited order to the audit store, with
  `rule_classification` and `final_classification` set.

Batch progress is tracked in a `batch_runs` row per audited order
(`pending` → `submitted` → `partial` → `complete`; `failed` is
terminal). Crashes mid-batch resume automatically on the next run.

## Safety boundaries

- The command reads HOSxP exports only. Output is written under
  `$BBA_DATA_DIR/audit/<run_id>/` and `$BBA_DATA_DIR/audit_runs/<run_id>/`.
  HOSxP tables are never touched.
- The command runs after the fact and stays silent. It does not send
  real-time alerts to clinicians. This is quality improvement (QI), not
  Software as a Medical Device (SaMD).
- A completed `run_id` cannot be silently overwritten. Re-runs of the
  same input always produce the same result in the same place. The
  invariant lives in the run-state store and stays that way.
